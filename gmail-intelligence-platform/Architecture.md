# Architecture & Design Document

## 1. System Architecture
The system follows a classic three-tier web application architecture designed for AI integration:
- **Frontend**: React (Vite) application providing the user interface for email management, chatting with the AI agent, and composing emails.
- **Backend**: Node.js + Express API handling business logic, user sessions, Google OAuth, and orchestrating calls to external APIs.
- **Database**: Supabase PostgreSQL with `pgvector` for structured data (emails, threads, labels, jobs) and vector embeddings (for semantic search).
- **External Services**: 
  - **Gmail API**: Used for securely syncing email data (readonly) and sending drafts.
  - **Google Gemini API**: Primary LLM used for summarization, drafting emails, thread-aware replies, and source-grounded RAG answering.
  - **NVIDIA NIM API**: Secondary model engine used specifically for structured tasks like categorization.

**Data Flow Overview**:
1. User authenticates via Google OAuth; backend securely handles tokens.
2. User initiates sync; backend fetches latest emails via Gmail API with pagination, parses MIME chunks, and upserts them into Supabase.
3. Upon ingestion, backend triggers embeddings generation using Gemini and categorizes emails using NVIDIA NIM.
4. When chatting, the backend embeds the query, retrieves relevant semantic chunks via pgvector, and injects them into the Gemini prompt (RAG) to ensure answers are strictly source-grounded.

## 2. Database Schema
We rely on a normalized PostgreSQL schema provided via Supabase to efficiently store and query hierarchical email threads. Key tables include:
- `users`: Core user accounts linked via Google OAuth.
- `gmail_accounts`: Securely stores the associated Gmail account metadata and backend-only OAuth tokens.
- `sync_state` & `sync_jobs`: Tracks pagination (`last_history_id`), success, and failure of incremental and full sync runs.
- `email_threads`: First-class representation of conversation threads. Holds aggregate metadata and thread-level summaries.
- `email_messages`: The core emails. Contains parsed headers, text/html bodies, `history_id`, and relates back to `email_threads`.
- `email_categories` & `email_summaries`: Stores generated AI insights to prevent repetitive processing.
- `email_embeddings`: Stores text chunks and their `vector(768)` embeddings generated by Gemini. This is queried by `match_email_embeddings` for RAG.
- `chat_sessions` & `chat_messages`: Tracks the user's conversational history with the AI to maintain context.

**Why pgvector?** It allows us to keep semantic data close to structured metadata. We can query embeddings localized explicitly to the authenticated `user_id`, guaranteeing cross-tenant data isolation directly at the database level.

## 3. AI Design
- **Summarization**: Handled by Gemini. Single emails are summarized based on raw body. Thread summaries chunk all emails chronologically to capture the full conversation arc.
- **Categorization**: Offloaded to a fast/efficient NVIDIA NIM model. It classifies emails into strict taxonomy (Newsletters, Job/Recruitment, Finance, Notifications, Personal, Work). Gemini acts as a fallback.
- **RAG Pipeline (Chat)**:
  1. User query is embedded using `gemini-embedding-001` (768 dimensions).
  2. `pgvector` performs a strict nearest-neighbor search (`<->`), strongly filtered by the `user_id`.
  3. The top chunks along with their metadata (Sender, Subject, Date) are injected into the Gemini context.
- **Source Clarity and Hallucination Prevention**: The system prompt given to Gemini explicitly forbids it from answering based on outside knowledge. If the context does not contain the answer, it outputs "I could not find this information in the synced emails". It is forced to produce a JSON response that pairs its answer with the exact `source` metadata items provided in the prompt context.

## 4. Gmail API Strategy
- **OAuth Scopes**: We request limited scopes (`gmail.readonly`, `gmail.send`) following the principle of least privilege.
- **Syncing and Pagination**: The initial MVP sync fetches the latest 50-100 emails using `nextPageToken`. For robust API interaction, we wrap all Google API calls in an exponential backoff utility to gracefully handle `429` Rate Limit limits.
- **Incremental Sync**: The `historyId` of the latest messages are tracked. Future iterations will fully leverage the `History API` to fetch only deltas.
- **Draft & Reply**: To prioritize user safety, AI replies are not sent automatically. We generate `drafts` respecting `In-Reply-To` and `References` headers, giving the user review capabilities before orchestrating the final `.send` command.

## 5. Tool & Technology Decisions
- **React + Vite**: Enables fast local dev and a responsive SPA needed for chat and email interfaces.
- **Node.js/Express**: Ideal for handling concurrent API orchestrations and asynchronous chunk processing/syncing without blocking execution.
- **Supabase**: Gives us a production-ready Postgres database out-of-the-box, alongside `pgvector` which prevents us from having to stand up an independent vector database like Pinecone.

## 6. Trade-offs & Limitations
- **Sync Limits**: For the MVP, we fetch a constrained number of recent emails rather than the entire historical inbox to accommodate API strictness and vector storage constraints.
- **Background Jobs**: Syncs run synchronously within the HTTP request for simplicity in setup. In a production system, this would be isolated to a background worker queue (e.g., BullMQ) or pub/sub architecture.
- **Token Storage**: OAuth tokens are physically encrypted across the wire and kept out of the frontend, but they are stored in plaintext securely in the backend-restricted Postgres instance. Production environments would manage this via a KMS or Supabase Vault.
- **Newsletter Deduplication**: Not implemented for the MVP due to time constraints, but the architecture (specifically `pgvector` mappings and semantic similarity) naturally paves the pathway for future similarity-based deduplication algorithms.
